# 研究生友好路线说明

## 题目方向

**跨形态机器人在测试时的快速适应：基于分层上下文建模的通用控制器**

这条路线的目标不是简单地把两个旧方法拼起来，而是回答一个更具体的问题：

> 当机器人形态不同、动力学变化、地形变化、甚至出现损伤时，能否让一个统一控制器在不重新训练的前提下，通过短历史快速推断当前上下文并恢复控制性能？

这条路线和 `RMA` 的在线自适应、以及 `MetaMorph` 的跨形态统一控制是强相关的，但你的新意应当放在：

- **分层上下文建模**
- **测试时快速推断**
- **跨形态条件下的适应**
- **上下文表示是否可迁移**

---

## 1. 研究目标

你可以把研究目标拆成三层：

1. **跨形态泛化**
   - 一个策略能控制不同模块组合、不同拓扑、不同 limb 数量的机器人。

2. **测试时适应**
   - 策略在面对未知摩擦、未知质量偏移、未知 terrain、局部损伤时，能够靠短历史快速恢复。

3. **上下文解耦**
   - 形态信息和时变适应信息最好不要混成一个大 latent，而是分开建模，再在控制器中融合。

如果你最后只能保住一个核心卖点，我建议保住第 2 和第 3 点，因为它们最像“新论文”的中心贡献。

---

## 2. 核心研究问题

你可以把问题定义成：

> 在 cross-embodiment control 中，哪些上下文应当作为静态条件显式给出，哪些上下文应当由历史在线推断，二者如何分层表示，才能在 unseen morphology 和 unseen dynamics 下保持鲁棒性与适应速度？

这里的关键是把“上下文”拆成两类：

- `z_m`：形态上下文
  - topology
  - limb 类型
  - joint 连接关系
  - body shape
  - gear / joint range / static hardware

- `z_a(t)`：适应上下文
  - terrain
  - friction
  - payload
  - motor degradation
  - local contact / slip
  - damage / disturbance

这个拆法有两个好处：

- 符合物理直觉
- 方便做消融，审稿人也容易理解

---

## 3. 总体方法

建议使用一个 **三段式系统**：

1. **形态编码器** `E_m`
   - 输入静态 morphology 信息
   - 输出 `z_m`

2. **历史适应编码器** `E_a`
   - 输入短历史 `h_t`
   - 输出 `\hat{z}_a(t)`

3. **Transformer 控制器**
   - 输入当前 limb tokens
   - 融合 `z_m` 和 `\hat{z}_a(t)`
   - 输出动作

---

## 4. 模型结构

### 4.1 文本示意图

```text
                +---------------------------+
                |   Static morphology info  |
                |  (shape, joints, gears,   |
                |   topology, hardware)     |
                +-------------+-------------+
                              |
                              v
                        +-----------+
                        |   E_m     |
                        | Morph Enc |
                        +-----+-----+
                              |
                             z_m
                              |
                              |
Current obs tokens            |             History window
(per-limb proprioception)     |         (o_{t-H+1:t}, a_{t-H+1:t-1})
             |                |                     |
             v                |                     v
      +--------------+        |              +--------------+
      | limb embed   |        |              |    E_a       |
      | per-limb x_i |        |              | Adapt Enc    |
      +------+-------+        |              +------+-------+
             |                |                     |
             v                |                \hat{z}_a(t)
      +-----------------------------+               |
      |  Transformer trunk           |<--------------+
      |  self-attention over limbs   |
      +--------------+--------------+
                     |
                     v
      +-----------------------------+
      | Fusion / Decoder             |
      | [token_i ; z_m ; \hat{z}_a] |
      +--------------+--------------+
                     |
                     v
                action per limb
```

### 4.2 更完整的流程图

```text
reset episode
   |
   v
collect current observation o_t
   |
   +--> static morphology branch -> z_m
   |
   +--> history buffer h_t -> E_a -> \hat{z}_a(t)
   |
   +--> current obs -> limb tokenization -> transformer trunk
                                    |
                                    v
                           fuse [token, z_m, \hat{z}_a]
                                    |
                                    v
                               action a_t
                                    |
                                    v
                            env step / reward
                                    |
                                    v
                         store (o_t, a_t, info)
                                    |
                                    +--> update history buffer
```

### 4.3 建议的融合方式

第一版建议用 **late fusion**，原因是：

- 最稳
- 最容易实现
- 最容易做消融
- 方便判断问题来自上下文表示，还是来自融合结构

实现形式可以是：

```text
decoder_input_i = concat(token_i, z_m, \hat{z}_a)
```

而不是一上来就做复杂的 cross-attention 或 MoE。

## 5. 推荐的实现步骤

### Step 1: 把问题定义清楚

先确认你的任务集合：

- 已知 morphology
- 未知 morphology
- 已知 dynamics
- 未知 dynamics
- 已知 terrain
- 未知 terrain

至少要能覆盖下面几种测试：

1. seen morphology + seen dynamics
2. unseen morphology + seen dynamics
3. seen morphology + unseen dynamics
4. unseen morphology + unseen dynamics

如果能加 terrain shift，会更完整。

### Step 2: 先做 teacher

Teacher 可以看到更强的 privileged 信息，例如：

- 静态 morphology 概要
- 摩擦
- 质量变化
- payload
- terrain slice
- 其他可获得的 state summary

Teacher 的任务是学会一个强策略，并产生一个“目标上下文” `z_a`.

### Step 3: 再做 student 的 history encoder

Student 只能看：

- 当前观测
- 历史窗口
- 历史动作

目标是从 history 中拟合 teacher 的上下文 latent：

```text
\hat{z}_a(t) = E_a(h_t)
```

这个阶段非常像 `RMA`，但你这里的背景是跨形态控制。

### Step 4: 把 student 接回控制器

把 `\hat{z}_a(t)` 融入 MetaMorph 风格的 transformer 控制器。

先用 late fusion 跑通：

- 当前 limb token
- `z_m`
- `\hat{z}_a(t)`

然后训练 PPO 或 imitation + PPO 的混合目标。

### Step 5: 做联合微调

当 student 已经能稳定预测上下文后，可以联合微调：

- morphology encoder
- adaptation encoder
- transformer trunk
- decoder

这一步的目标是让整个系统更适应闭环控制。

---

## 6. 完整文字流程

你可以把系统流程理解为以下闭环：

1. 环境 reset，得到机器人当前形态、初始状态和任务信息。
2. 系统读取形态相关的静态信息，送入 morphology encoder，得到 `z_m`。
3. 系统维护一个历史缓冲区，记录最近 `H` 步的观测和动作。
4. 历史缓冲区输入 adaptation encoder，得到 `\hat{z}_a(t)`。
5. 当前时刻的 proprioceptive observation 被切分成 per-limb token。
6. per-limb token 进入 transformer trunk，得到每个 limb 的中间表示。
7. 中间表示与 `z_m`、`\hat{z}_a(t)` 融合。
8. decoder 生成动作。
9. 环境执行动作，返回 reward、done 和新状态。
10. 历史缓冲区更新，进入下一步循环。

这条流程的关键不是“预测一个 latent”，而是**让 latent 成为控制循环的一部分**，也就是在线 context formation。

---

## 7. 训练路线建议

### 路线 A: 最稳

1. 先训 teacher policy
2. 冻结 teacher，训 student history encoder
3. 用 student latent 驱动控制器
4. 最后联合微调

适合你现在的阶段，因为最容易拆分问题。

### 路线 B: 更像顶会论文

1. 先训统一 morphology policy
2. 引入 context factorization
3. 引入 adaptation latent
4. 做 test-time adaptation benchmark

这条线更适合最终投稿版本。

### 路线 C: 更激进

1. 把 context 做成可路由的模块
2. 让 context 决定激活哪些参数或哪些 limb attention
3. 再加入 history-based adaptation

这条线更强，但实现和调参难度更高。

## 8. 可参考的评价标准

顶会级项目不能只看平均 reward，建议至少覆盖以下几类指标。

### 8.1 控制性能指标

- episodic return
- success rate
- forward velocity tracking error
- energy efficiency
- fall rate / termination rate

### 8.2 泛化指标

- unseen morphology return
- unseen terrain return
- unseen dynamics return
- unseen morphology + unseen dynamics combined return

### 8.3 适应指标

- adaptation speed
  - 进入新环境后多少步恢复到稳定控制
- recovery area under curve
  - 扰动后回报恢复曲线下面积
- performance drop
  - 扰动前后性能下降幅度

### 8.4 表示质量指标

- `z_m` 是否跨 morphology 一致
- `\hat{z}_a(t)` 是否能反映 dynamics shift
- latent interpolation 是否对应平滑行为变化
- latent probing
  - 能否从 latent 预测摩擦、质量、损伤类型等

### 8.5 结构性分析指标

- attention map 可视化
- limb-level response analysis
- history length sensitivity
- context ablation

## 9. 顶会级项目通常还需要什么

如果你想把它做成顶会稿子，下面这些是基本配套。

### 9.1 一个清晰的 benchmark

最好不是只在一个环境上测，而是有系统变化：

- 多 morphology
- 多 terrain
- 多 dynamics
- 最好还有损伤或故障

### 9.2 强 baseline

至少要对比：

- 标准 MetaMorph
- RMA-style history latent
- 只做 morphology conditioning
- 只做 adaptation latent
- 你的分层上下文方法

### 9.3 消融实验

必须要有：

- 去掉 `z_m`
- 去掉 `z_a`
- `z_m` 和 `z_a` 合并成一个 latent
- 不同 history 长度
- late fusion vs token fusion
- privileged teacher vs student only

### 9.4 分析图

最好准备：

- 适应曲线图
- latent t-SNE / UMAP
- 不同形态下的 attention 可视化
- 扰动前后恢复对比图

### 9.5 清楚的贡献点

建议贡献点写成三条左右：

1. 提出分层上下文建模，把 morphology 和 dynamics context 分开。
2. 提出跨形态 test-time adaptation 机制。
3. 在统一 benchmark 上验证 unseen morphology / unseen dynamics 的泛化和恢复能力。

---

## 10. 你现在最应该做什么

如果你现在要推进项目，我建议顺序是：

1. **定任务**
   - 先确定你要在哪个仿真环境做。

2. **定上下文**
   - 明确 `z_m` 和 `z_a(t)` 各自包含什么。

3. **做最小原型**
   - morphology encoder + history encoder + late fusion transformer。

4. **先跑通 teacher/student**
   - 不要一开始就上复杂 routing。

5. **做基线对比和消融**
   - 这是将来能不能投稿的关键。

---

## 11. 一个适合研究生推进的里程碑

### 第 1 阶段

- 复现或跑通 baseline
- 明确 observation 设计
- 明确 morphology / dynamics 的划分

### 第 2 阶段

- 实现 teacher policy
- 实现 history encoder
- 实现 latent regression

### 第 3 阶段

- 接回 transformer policy
- 做 test-time adaptation
- 跑 unseen morphology / unseen dynamics

### 第 4 阶段

- 做消融
- 做可解释性分析
- 形成论文叙事

---

## 12. 一句话总结

这条路线最适合的表达不是“我把 RMA 和 MetaMorph 拼起来”，而是：

> 我们把跨形态控制中的静态形态上下文和时变适应上下文分层建模，并通过测试时的在线上下文形成机制，实现机器人在未知形态、未知动力学和未知地形下的快速适应。
