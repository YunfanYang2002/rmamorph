# 2026-05-28 11:06

摘要：这一版把 `MetaMorph` 从“纯跨形态控制器”推进成了“带显式上下文的跨形态快速适应原型”。当前已经接入 `static_context`、`adaptive_context`、`history_context` 三条信息通道，并为后续的 teacher/student / online adaptation 预留了完整接口。

## 1. 本次更新做了什么

这次代码更新的目标，不是把系统一次性做成完整的 `RMA`，而是先把“跨形态控制 + 在线上下文形成”这条路线的最小闭环搭起来。

### 1.1 新增了上下文通道

除了原有的 `proprioceptive`、`edges`、`obs_padding_mask`、`act_padding_mask` 以外，模型现在还能额外看到三类上下文：

- `static_context`
- `adaptive_context`
- `history_context`

它们分别表示：

- `static_context`：形态和硬件的相对静态信息，例如 body shape、joint range、gear、friction。
- `adaptive_context`：当前时刻可见的适应性信息，例如姿态、速度、局部动力学状态。
- `history_context`：最近一段时间的观测和动作历史，为后续 student-style adaptation 提供输入。

### 1.2 新增了上下文编码器

`TransformerModel` 不再只依赖 limb token 的 self-attention，而是可以先把上下文压成较小的 latent，再在 decoder 前做 late fusion。

新增的编码分支包括：

- `static_context_encoder`
- `adaptive_context_encoder`
- `history_context_encoder`

这些编码器的职责不是直接输出动作，而是把不同来源的信息压缩成统一 latent，让控制器更容易消费。

### 1.3 新增了 history buffer

agent 现在维护一个历史缓存区，记录最近若干步的：

- `proprioceptive observation`
- `action`

这个缓存会被压平后写入 `history_context`，作为 student-style adaptation 的输入。

### 1.4 新增了 history context wrapper

为了让 PPO buffer 和 observation space 都知道 `history_context` 的存在，环境里新增了一个 wrapper，用来预留固定大小的 `history_context` 占位。

这样做的好处是：

- 训练流程不用大改
- buffer 可以自动保存这个字段
- 后面接 history encoder 时不容易出现 shape 不一致的问题

## 2. 这些结构为什么有意义

### 2.1 `static_context` 的意义

`static_context` 的核心意义是把“形态是什么”显式告诉模型。

如果没有这条通道，模型只能从 limb token 里自己猜 morphology，这会让很多形态差异和控制差异混在一起。

有了 `static_context` 以后，模型就更容易区分：

- 哪些差异来自机器人结构
- 哪些差异来自当前状态或环境

这对跨形态控制非常重要，因为不同 morphology 的策略不应该完全依赖同一个共享 MLP 去“硬凑”。

### 2.2 `adaptive_context` 的意义

`adaptive_context` 的意义是把当前可见的物理状态和局部环境线索显式编码出来。

在机器人任务里，很多变化并不是静态的，例如：

- 摩擦变化
- payload 变化
- 接触状态变化
- terrain 变化

如果这些信息完全埋在 observation 里，模型会更难学出稳定规律。把它们单独编码以后，模型更容易学习“当前环境状态如何影响动作分布”。

### 2.3 `history_context` 的意义

`history_context` 是这次更新里最关键的结构性变化。

它的作用不是“再加一点输入”，而是给模型一个时间维度上的记忆入口。这样模型就不只是看当前一步，而是可以结合最近几步轨迹推断：

- 当前是不是接近失稳
- 地形是不是更复杂
- 机器人是不是逐渐偏离目标状态
- 最近动作有没有引起明显姿态漂移

这一步的研究意义在于：策略开始从“瞬时映射”走向“有记忆的上下文形成器”。

## 3. 这版理想中应该带来的效果

### 3.1 对训练过程的影响

- 更容易学稳
- 不容易因为 morphology 差异太大而训练发散
- 收敛曲线更平滑
- 早期探索阶段的回报波动更小

### 3.2 对控制结果的影响

- 平均 `Reward` 更高
- `episode length` 更长
- `forward velocity` 更稳定
- 能量或控制代价不要明显恶化

### 3.3 对泛化能力的影响

- 在 unseen morphology 上不至于完全崩掉
- 在 unseen dynamics 上恢复更快
- 在 terrain 变化下更容易重新找到可行动作

### 3.4 对后续研究的意义

- `static_context` 和 `history_context` 的接口已经固定
- 后面只需要继续补 teacher / student supervision
- 这意味着当前代码不是一次性拼接，而是可以持续迭代的研究原型

## 4. 当前结果应该怎么理解

如果训练结果里 `Reward` 从较低值明显涨到较高值，说明至少有三件事成立：

- 新结构没有破坏训练管线
- 当前上下文通道是可用的
- 策略确实学到了和环境交互相关的有效控制模式

但需要注意：

- 这还不等于已经严格证明“哪个模块单独带来了提升”
- 真正的模块贡献还需要 baseline、消融实验和不同 context 配置的对比

换句话说：

- **现在的结果说明方向是对的**
- **还不能直接把结论写成“模块已经被严格证明有效”**

## 5. 给后来阅读这份日志的人一个最短理解路径

如果你第一次看这个项目，建议按下面顺序理解：

### 5.1 先看数据流

先回答三个问题：

- observation 里现在有哪些 key
- 哪些 key 是静态的
- 哪些 key 是历史相关的

只要这一步清楚了，后面看模型就会轻松很多。

### 5.2 再看是谁生成这些 key

这个系统可以拆成三部分：

- 环境负责生成原始状态
- wrapper 负责补齐结构化字段
- agent 负责维护 history buffer

这样你就能理解为什么有些东西适合放在环境里，有些东西适合放在 agent 里。

### 5.3 最后看模型怎么消费这些 key

模型层做的事情很简单：

- 先把不同 context 编成 latent
- 再和 limb token 一起送进 decoder

如果记住这一点，就不容易被 `static_context`、`adaptive_context`、`history_context` 这些名字绕晕。

摘要：把运行模式的命名和 TensorBoard 目录统一成“按模式分层”的形式，避免后面 baseline、teacher、student、hybrid 的日志混在一起。现在 `cfg.get_run_mode_tag()` 会自动生成模式标签，TensorBoard 会写到 `output/tensorboard/<mode_tag>/...`。

## 1. 本次更新做了什么

- 新增了统一的运行模式标签生成函数：`cfg.get_run_mode_tag()`
- 新增了统一的 run name 生成函数：`cfg.get_run_name()`
- TensorBoard 目录改成按模式分层
- 为后续新增功能留了一个明确的命名入口，避免每次都手工改日志路径

## 2. 命名规则是什么

当前模式标签的生成逻辑是：

- `MODEL.CONTEXT_MODE = none` 时，模式名为 `baseline`
- `MODEL.CONTEXT_MODE = teacher` 时，模式名为 `teacher`
- `MODEL.CONTEXT_MODE = student` 时，模式名为 `student`
- `MODEL.CONTEXT_MODE = hybrid` 时，模式名为 `hybrid`

如果启用了 `HistoryContextWrapper`，模式名后面会再加上 `_history`。例如：

- `baseline_nohistory`
- `teacher_history`
- `student_history`
- `hybrid_history`

这意味着 TensorBoard 目录会自动变成类似：

- `output/tensorboard/baseline_nohistory/...`
- `output/tensorboard/student_history/...`

## 3. 为什么这样更方便

- 看目录名就知道当前实验处于什么模式
- baseline、teacher、student 的曲线不会混在一起
- 后面如果新增功能，只要更新这一个模式入口，日志命名就能一起跟着变
- 对代码能力一般的读者更友好，因为先看文件夹名就能知道自己在看什么实验

## 4. 对后续工作的建议

如果后面再加新功能，比如新的 context、routing、memory、aux loss，建议继续遵守这个规则：

- 先把功能映射成明确的模式标签
- 再让 `cfg.get_run_mode_tag()` 或相关命名函数统一输出
- 最后让 TensorBoard 和输出目录都跟着这个标签走
